Data Pipeline & Knowledge Base

A reliable knowledge base prevents LLM hallucinations. I constructed a structured database from Paula's Choice Ingredient Dictionary, employing a specialized dual-chunking strategy for optimal Retrieval-Augmented Generation (RAG).

Dual-Chunking Strategy

Summary Chunks
1 per ingredient. Contains name, categories, safety. Optimized for fast metadata filtering.
Detail Chunks
0-N per ingredient. ~400 char segments with overlap. Preserves context for deep QA.

System Design

Modular Architecture

Two independent AI pipelines share a single Weaviate vector database and are connected by a Scan Context Injection bridge — when a product is scanned in Pipeline 1, its ingredient state is automatically passed into Pipeline 2 queries, enabling follow-up questions like "Is any of these unsafe for sensitive skin?"

Offline Ingestion

The 2,509-ingredient KB is chunked into 7,459 vectors and indexed into Weaviate once. Only Pipelines 1 & 2 run at inference time.

Dual Pipeline

Pipeline 1 handles vision-to-safety. Pipeline 2 handles natural language Q&A. They run independently but share session state.

Context Bridge

After a product scan, the ingredient list is injected into the RAG Prompt Builder — enabling follow-up questions without re-scanning.

RAG & NLP Architecture

Hybrid Search Strategy

Weaviate is configured with an alpha = 0.35 weighting. This deliberately prioritizes keyword matching (BM25) over semantic similarity.

Why? Favoring exact term matching over broad semantic search yields higher precision for specific chemical names.

Query Classification Router

  • Ingredient-Specific "What does niacinamide do?" → Direct lookup bypasses LLM latency.
  • Open-Ended "What ingredients help with acne?" → Triggers full hybrid semantic search.

Retrospective

Limitations & Future Work

Evaluating the current capabilities of the multimodal AI system and identifying pathways for future enhancements.

🚧 Limitations

  • ✦
    OCR Reliability Small fonts, curved surfaces, and reflections degrade text recognition quality and lead to extraction errors.
  • ✦
    Name Normalization Mapping noisy OCR outputs with spelling errors and abbreviations to standardized names remains imperfect.
  • ✦
    Quantitative Evaluation Absence of labeled datasets means evaluation primarily relies on qualitative manual testing.
  • ✦
    Open-Ended Query Coverage Because the database focuses on individual ingredients, broad dermatological questions may yield incomplete answers.

🚀 Future Work Roadmap

  • ✦
    Improving OCR Robustness Focus on reliability under real-world conditions via image preprocessing and specialized detection models.
  • ✦
    Enhanced Normalization Incorporate synonym dictionaries and lightweight models to accurately map noisy OCR outputs.
  • ✦
    Evaluation Dataset Construct a labeled dataset of cosmetic labels to enable quantitative benchmarking for system reliability.
  • ✦
    Knowledge Base Expansion Integrate additional cosmetic and dermatology databases to improve retrieval coverage and analysis depth.

Conclusion

Cosmetic Ingredient Analyzer demonstrates how computer vision, NLP, and retrieval systems can be integrated into a practical AI product. By combining ingredient label scanning, safety classification, and knowledge-grounded question answering, the system transforms complex cosmetic ingredient information into accessible insights for everyday users, highlighting the potential of multimodal AI to improve consumer transparency.
